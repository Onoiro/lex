# Lex: Local-First Architecture Roadmap

## Текущее состояние
FastAPI + SQLAlchemy + SQLite + Jinja2 (server-rendered). Вся бизнес-логика — серверная. API-ключ Yandex — в `.env` на сервере. Безопасность: Basic Auth, CSRF, rate limiting, валидация ввода — всё серверное.

## Целевое состояние
Local-first: словарь, SRS, настройки — локально на устройстве, без регистрации и без сети. Интернет нужен только для перевода через тонкий proxy (ключ остаётся серверным). Дистрибуция: RuStore + AppGallery (Android, Capacitor), Desktop (Tauri).

## Фазы

### Phase 0: Architecture & Tech Stack
- **Цель:** Выбрать клиентский фреймворк, движок хранения и структуру нового проекта.
- **Вход:** Анализ текущей кодовой базы (модель, SRS-логика, шаблоны, i18n).
- **Выход:** Решение: фреймворк (React/Vue/Svelte), хранилище (IndexedDB / SQLite WASM / OPFS), CSS-подход, скелет проекта.
- **Зависимости:** Нет.
- **Статус:** [x] завершена

### Phase 1: Translate Proxy
- **Цель:** Вынести интеграцию Yandex Translate API в отдельный легковесный proxy-сервис, скрывающий API-ключ.
- **Вход:** `services/translator.py`, `services/cache.py`, `security/rate_limiter.py`.
- **Выход:** Минимальный proxy (FastAPI или serverless) с эндпоинтами `/translate` и `/languages`, rate limiting, кэш переводов. Деплоится независимо от клиентского приложения.
- **Зависимости:** Нет (может идти параллельно с Phase 0).
- **Статус:** [x] завершена

### Phase 2: Client-side Data Layer
- **Цель:** Портировать модель Word и постоянное хранилище на клиент.
- **Вход:** `models.py` (сущность Word: 11 полей), `database.py`.
- **Выход:** Клиентская схема хранения, повторяющая текущую модель Word; CRUD-операции (add, delete, list, update), проверка дубликатов.
- **Зависимости:** Phase 0 (выбор хранилища).
- **Статус:** [x] завершена

### Phase 3: Client-side Domain Logic (SRS & Stats)
- **Цель:** Портировать алгоритм интервальных повторений, выбор слов и трекинг статистики на клиент.
- **Вход:** Функции из `main.py`: `_apply_review_result` (SM-2), `_pick_weighted_word` (взвешенный рандом), `_update_response_time` (best/avg), `_pick_random_direction`.
- **Выход:** Клиентский доменный модуль: SRS-движок, взвешенный выбор слова, трекер времени ответа, выбор направления. Покрыт unit-тестами.
- **Зависимости:** Phase 2 (работает поверх data layer).
- **Статус:** [x] завершена

### Phase 4: Client-side UI & i18n
- **Цель:** Переписать все экраны как клиентские компоненты с полной поддержкой i18n.
- **Вход:** Jinja2-шаблоны (index, add, review, dictionary, settings), `i18n/` (en.json, ru.json, languages.py), inline-JS из шаблонов (таймер, авто-ответ, пауза, prefetch). Валидация из `security/validators.py`.
- **Выход:** Полноценное клиентское приложение: главная (дашборд), добавление слова (кнопка перевода → proxy), повторение (таймер, авто-ответ, пауза, prefetch), словарь (таблица, поиск, удаление), настройки (селекторы языков). i18n портирован. Валидация ввода — на клиенте.
- **Что исчезает:** Basic Auth, CSRF — не нужны (local-first, нет серверных форм).
- **Зависимости:** Phase 1 (proxy для перевода), Phase 2 (данные), Phase 3 (доменная логика).
- **Статус:** [x] завершена

### Phase 5: PWA
- **Цель:** Сделать приложение устанавливаемым и полностью работающим офлайн.
- **Вход:** Клиентское приложение из Phase 4.
- **Выход:** PWA с service worker (офлайн-шелл + кэш ассетов), web manifest, иконки, install prompt. Перевод gracefully деградирует при отсутствии сети.
- **Зависимости:** Phase 4.
- **Статус:** [x] завершена

### Phase 6: Android (Capacitor)
- **Цель:** Упаковать PWA как нативное Android-приложение для RuStore и AppGallery.
- **Вход:** PWA из Phase 5.
- **Выход:** Capacitor-проект, настроенные permissions, сборка APK/AAB, store-ready ассеты (иконки, скриншоты), конфигурации публикации в RuStore и AppGallery.
- **Зависимости:** Phase 5.
- **Статус:** [x] завершена

### Phase 7: Desktop (Tauri)
- **Цель:** Упаковать клиентское приложение как нативное десктоп-приложение.
- **Вход:** PWA из Phase 5.
- **Выход:** Tauri-проект, нативные инсталляторы для Windows (MSI/NSIS), macOS (DMG), Linux (AppImage/deb). Конфигурация автообновления.
- **Зависимости:** Phase 5 (может идти параллельно с Phase 6).
- **Статус:** [x] завершена

### Phase 8: Migration & Decommission
- **Цель:** Позволить существующим пользователям перенести словарь из старого серверного приложения; вывести из эксплуатации старый FastAPI-сервер.
- **Вход:** Все предыдущие фазы завершены.
- **Выход:** Инструмент экспорта из старого SQLite → импорт в клиентское хранилище. Обновлённые KODA.md/README. Старый сервер сведён до proxy-only или выведен из эксплуатации.
- **Зависимости:** Все предыдущие фазы.
- **Статус:** [ ] не начата

## Параллелизм

```
Phase 0 ─┬──> Phase 2 ──> Phase 3 ─┐
          │                         ├──> Phase 4 ──> Phase 5 ─┬──> Phase 6
Phase 1 ──┘                         │                          └──> Phase 7
                                     │                                   │
                                      └────────────────> Phase 8
```

- Phase 0 и Phase 1 — независимы, стартуют одновременно.
- Phase 2 и Phase 3 — последовательны (данные → логика).
- Phase 4 требует 1+2+3.
- Phase 6 и Phase 7 — параллельны после Phase 5.
- Phase 8 — финальная, после всех.
