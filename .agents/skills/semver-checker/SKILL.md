# Semver Checker — проверка версии перед коммитом

## Когда срабатывать

При любом запросе на коммит: «закоммить», «сделай commit», «зафиксируй изменения», «commit», «подготовь коммит» и т.п. **До** формирования сообщения коммита и выполнения `git commit`.

## Что делает навык

1. Анализирует изменения (`git diff`) и определяет тип semver-бампа.
2. Если бамп нужен — обновляет версию во **всех** файлах проекта.
3. Включает информацию о версии в сообщение коммита.

## Шаг 1: Сбор информации

Выполни параллельно:
- `git diff --staged` — если есть staged-изменения (предпочтительно).
- `git diff HEAD` — если staged пуст, но есть unstaged-изменения.
- Прочитай текущую версию из `pyproject.toml` (строка `version = "x.y.z"`).

## Шаг 2: Определение типа изменения

Проанализируй файлы и характер изменений в diff. Используй следующие правила:

### Major (x.0.0) — только для 0.x → 1.0 или уже 1.x+ при breaking change
- Переход на новую архитектуру (например, local-first).
- Удаление существующей функциональности.
- Несовместимые изменения API или формата данных.
- Фундаментальное изменение стека технологий.

### Minor (0.x.0 или 1.x.0)
- Новые функции, новые страницы, новые эндпоинты.
- Существенное расширение возможностей.
- Обратная совместимость сохранена.

### Patch (0.0.x или 1.0.x)
- Исправление багов.
- Небольшой рефакторинг.
- Обновление зависимостей.
- Улучшение UX без новой функциональности.

### None (версия не меняется)
- Только документация (README, KODA.md, комментарии).
- Только тесты.
- Только CI/CD конфигурация.
- Только форматирование/стиль кода без логических изменений.
- Изменения в `.gitignore`, линтер-конфигах.

**Важно:** Если сомневаешься между minor и patch — выбирай patch. Между major и minor — выбирай minor. Бамп major должен быть обоснованным.

## Шаг 3: Проверка рассинхрона версий

Сравни версии во всех файлах. Если они расходятся — это баг, который нужно исправить даже при `none` (синхронизировать на текущую версию из `pyproject.toml`).

## Шаг 4: Обновление версии (если бамп нужен)

Если тип изменения — не `none`, вычисли новую версию и обнови **все** файлы ниже.

### Файлы с версией

| Файл | Что менять | Формат |
|---|---|---|
| `pyproject.toml` | `version = "x.y.z"` | Строка в кавычках |
| `uv.lock` | `version = "x.y.z"` (строка `name = "lex"` чуть выше) | Строка в кавычках |
| `client/package.json` | `"version": "x.y.z"` | JSON-строка |
| `client/src-tauri/tauri.conf.json` | `"version": "x.y.z"` | JSON-строка |
| `client/android/app/build.gradle` | `versionCode N` и `versionName "x.y.z"` | versionCode — целое, +1 к предыдущему |

### Правила для versionCode (Android)

- `versionCode` — монотонно возрастающее целое число.
- Всегда увеличивай на 1 от текущего значения (не вычисляй из versionName).
- Текущее значение читай из `build.gradle` перед изменением.

### Пример обновления

Текущая версия `0.11.7`, тип изменения — major:
- Новая версия: `1.0.0`
- `pyproject.toml`: `version = "1.0.0"`
- `uv.lock`: `version = "1.0.0"` (в секции `name = "lex"`)
- `client/package.json`: `"version": "1.0.0"`
- `client/src-tauri/tauri.conf.json`: `"version": "1.0.0"`
- `client/android/app/build.gradle`: `versionCode 12`, `versionName "1.0.0"`

## Шаг 5: Проверка согласованности

После обновления всех файлов, перечитай их и убедись, что версия одинакова везде. Если хотя бы один файл не совпадает — исправь.

## Шаг 6: Информация в сообщении коммита

Включи в сообщение коммита (в конце, после основного описания) строку:

```
Bump version: 0.11.7 → 1.0.0 (major)
```

Если версия не менялась, но был исправлен рассинхрон — добавь:

```
Sync version across files: 0.11.7
```

Если изменений версии нет и рассинхрона нет — ничего не добавляй.

## Пограничные случаи

- **Версия уже обновлена в diff**: Если в diff уже есть изменения версии во всех файлах и они согласованы — не дублируй обновление, но проверь согласованность.
- **Только proxy-изменения**: Proxy не имеет отдельной версии, используется `pyproject.toml`. Применяй стандартные правила.
- **Несколько коммитов подряд**: Каждый коммит анализируется независимо. Если предыдущий коммит уже сделал minor-бамп, а этот — patch-фикс, бампай patch от новой версии.
- **Pre-release**: Если текущая версия содержит суффикс (например `1.0.0-rc.1`), учитывай его при вычислении новой версии.
